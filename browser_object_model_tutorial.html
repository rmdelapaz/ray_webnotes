The Browser Object Model (BOM)

Browsers: They’re The BOM Dot Com!

If the Internet exists, but there’s no way to browse it, does it even really exist? Unless you’ve been living under a rock for the past couple of decades, you should know what a browser is, and you probably use it multiple times a day. Browsers are something most people take for granted, but behind the scenes is a complex structure working to display information to users who browse the Web.

Web developers rely on browsers constantly. They can be your best friend or your worst enemy. (Yes, we’re looking at you, IE!) Spending some time learning about browsers will help you get a higher-level understanding of how the Web operates, how to debug, and how to write code that works across browsers. In this reading, we’ll learn about the BOM (Browser Object Model), how it’s structured, and how it differs from the DOM (Document Object Model).

The DOM vs. the BOM

By now, you’ve learned about the DOM, or Document Object Model, and that it contains a collection of nodes (HTML elements), that can be accessed and manipulated. In essence, the document object is a Web page, and the DOM represents the object hierarchy of that document.

How do we access a document on the Web? Through a browser, of course! If we took a bird’s-eye view of the browser, we would see that the document object is part of a hierarchy of browser objects. This hierarchy is known as the BOM, or Browser Object Model.

Analogy: Think of the BOM as a toolbox that the browser provides to help you interact with everything outside the HTML document. While the DOM is like the house you live in (the web page), the BOM is like the surrounding landscape—roads, mailboxes, and the utility systems (water, power, internet) that keep everything running smoothly.

The Window Object: The Chief Browser Object

The chief browser object is the window object, which contains properties and methods we can use to access different objects within the window. These include:

  • window.navigator - Returns a reference to the navigator object, which tells you information about the browser and the operating system.
  • window.screen - Returns a reference to the screen object associated with the window, giving you details about the user’s display resolution.
  • window.history - Returns a reference to the history object, allowing you to navigate back and forward through the browser’s history.
  • window.location - Gets/sets the location, or current URL, of the window object. It’s like the address on the envelope—this tells you where you are.
  • window.document - Returns a reference to the document that the window contains. You can also use document directly.

Metaphor: Think of the window object as the command center of the browser. Just like a pilot has a dashboard with controls to navigate a plane (altitude, speed, direction), the window object has controls to navigate and interact with the web environment (location, history, screen).

Shortening the Window Object

Note how we can shorten window.document to document. For example, the document in document.getElementById(‘id’) actually refers to window.document. All of the methods above can be shortened in the same way.

The Browser Diagram

We started in the DOM, and we stepped outside it into the BOM. Now, let’s take an even higher view of the browser itself. Imagine a diagram depicting a high-level structure of the browser:

  • User Interface (UI): This is the browser interface that users interact with, including the address bar, back and forward buttons, bookmarks menu, etc. It includes everything except for the requested page content. Think of it as the cockpit of a plane where you, as the pilot, control the flight (web experience).
  • Browser Engine: Manages the interactions between the UI and the rendering engine. Like a translator, it ensures the right data gets to the right place.
  • Rendering Engine: Displays, or renders, the requested page content. If the requested content is HTML, it will parse HTML and CSS and render the parsed content. This is where the magic of transforming code into a visual page happens—like baking a cake from a recipe.
  • Networking: Handles network calls, such as HTTP requests. Every time you load a web page, it’s like placing an order online; the browser sends a request (order) and waits for a response (the package).
  • JavaScript Interpreter: Parses and executes JavaScript code. This is what makes your web page dynamic, responsive, and interactive.
  • UI Backend: Used for drawing basic widgets like combo boxes and windows; uses operating system user interface methods.
  • Data Storage: The persistence of data stored in the browser, such as cookies, local storage, and session storage. This is like a fridge—storing data that can be reused later.

  • Understanding how the BOM differs from the DOM, focusing more on the browser environment.
  • The window object and related methods, acting as the central hub for browser-related actions.
  • A higher-level view of the browser's inner workings, including how it processes, renders, and manages interactions with web content.
